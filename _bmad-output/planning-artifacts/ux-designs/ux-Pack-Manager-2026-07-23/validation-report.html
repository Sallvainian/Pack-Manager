Usability review

review-usability.md

The mental model is strong, but several consequential actions have ambiguous scope or consequences.

High

Plan items cannot be edited directly in the sidecar

Update Everything provides no defined way to remove one Package or Manager self-update.

Fix: Make every staged item removable, define In Plan as a reversible state, update counts immediately, and return Change Plan focus to the editable plan.

High

Header Checkbox scope is ambiguous

“Eligible visible rows” could mean rendered rows or the complete active-filter result.

Fix: Apply it to every eligible Package matching the active filter, including virtualized rows; show the exact count and mixed state.

High

Confirmation-disabled action is unclear

Confirm # Updates implies another confirmation even when the action executes immediately.

Fix: Show Confirmation is off, link to Settings, expand commands, and relabel the action Run # updates.

High

Abnormal Activity controls are undefined

Cancel, skip, wait, timeout, and unexpected command interaction have states but no complete control contract.

Fix: Define when each safe action appears, what it affects, and an explicit Interaction required blocked state for unexpected prompts.

High

Retry conflicts with one-entry History semantics

Fix: Each confirmed attempt creates a new History entry linked to its source retry; the original result remains unchanged.

Medium

Activity presentations can conflict

Define one shared live model, replay behavior during active work, and a Back to live activity path.

Medium

Retry may displace unread Results

Define the transition and preserve the completed diagnosis in History with a return path.

Medium

Unavailable states lack standard explanations

Explain excluded/unavailable Packages and use plain language such as Managed through mise.

Medium

Application update discovery lacks a canonical location

Use Settings → Pack-Manager updates plus one restrained application-level notice.

Medium

Settings persistence is undefined

Choose immediate saving with rollback or one consistent Apply/Cancel model.

Low

Results dismissal has two names

Choose Done consistently.

Low

History updated/total can overstate success

Use verified wording such as 10 of 12 verified · 2 failed.

Accessibility review

review-accessibility.md

The accessibility direction is strong, but implementation could still lose focus or expose incomplete virtualized content unless the remaining details become normative.

High

Focus boundary token is too weak

borderStrong measures only 2.33:1 against surface and no separate focus token is defined.

Fix: Use accent or a dedicated token with a separated minimum 2px ring and distinguish focus from selection.

High

Virtualized Package-table operation is incomplete

Table semantics, bounded Tab behavior, arrow navigation, stable focus, position metadata, and final-row reachability are not defined.

Fix: Add one normative table interaction model and verify the packaged app with VoiceOver and 101 rows.

High

Lifecycle focus transitions are undefined

Fix: Define initial dialog focus, final-confirm focus, stable Activity focus, Results announcement/focus, and focus restoration after Results closes.

High

Fixed panes conflict with 200% zoom

Fix: Add a high-zoom/narrow mode that collapses navigation and makes Plan/Activity/Results a full-workspace or stacked surface; test 100%, 150%, and 200%.

Medium

Live-announcement behavior is underspecified

Define one atomic channel for meaningful plan events, suppress duplicate notification speech, and never announce output lines or progress ticks.

Medium

Explanatory-disabled semantics are contradictory

A native disabled Checkbox cannot receive focus or activation. Use an unavailable semantic state without native disabled, attach a persistent accessible description, and keep activation inert.

Recommended resolution order

  1. Resolve the five high-impact workflow ambiguities and four high-impact accessibility contracts.
  2. Resolve coexistence, persistence, unavailable-state, announcement, and explanatory-disabled details.
  3. Standardize Done and verified History wording.
  4. Render the remaining authoritative mockups only after the high findings are committed.

Steps 1–4 and finalization are complete.
